PrintDesk - Web-to-Print Portal

מסמך מוצר, שלבים ועיצוב

PrintDesk הוא פורטל הדפסה למרחבי עבודה: משתמש מעלה PDF באתר, הענן שומר ומנהל את העבודה, וסוכן Windows מקומי מושך החוצה את העבודה ומדפיס במדפסת הפיזית. המחשב המקומי תמיד יוזם תקשורת אל הענן.

עקרונות מוצר

מה בונים ולמה

המוצר מחליף תהליך ידני של שליחת מסמכים למנהל המרחב בתהליך שקוף, מוגבל, מאובטח וקל לתפעול.

ענן

ניהול עבודות

  • טופס העלאת PDF למשתמשים.
  • שמירת קבצים ב-Supabase Storage פרטי.
  • טבלת עבודות עם סטטוס ברור לכל שלב.
סוכן מקומי

הדפסה בטוחה

  • מחשב Windows מקומי מושך עבודות מהענן.
  • אין פתיחת פורטים ואין חשיפת מדפסת לרשת.
  • הדפסה שקטה של PDF דרך SumatraPDF.
מגבלות

MVP ממוקד

  • PDF בלבד.
  • מגבלת גודל, עותקים ותצוגת סטטוס.
  • אין הדפסה אנונימית בלתי מוגבלת.

דיאגרמת ארכיטקטורה

הענן לא נכנס פנימה, המקומי יוצא החוצה

כל התקשורת הרגישה נעשית ביוזמת המחשב המקומי מול Supabase. אין תלות ברשת פנימית פתוחה.

סטטוסים

מפת מצב מלאה

כל עבודה חייבת להיות במצב ברור, כדי שמשתמש, אדמין וסוכן הדפסה יבינו מה קורה.

pendingממתין לאישור מנהל.
approvedמאושר וממתין בתור לסוכן.
claimedהסוכן תפס את העבודה.
downloadingהקובץ יורד למחשב המקומי.
printingהקובץ נשלח למדפסת.
printedהפקודה הסתיימה בהצלחה.
failedנכשל ויש להציג סיבה.
cancelledבוטל לפני הדפסה.
rejectedנדחה על ידי אדמין.

טאבים

כל שלבי המימוש

הטאבים מפרקים את הפרויקט לשלבים, בלי לקפוץ ישר למוצר ענק מדי.

Phase 0

Project Setup

תכולה

  • Monorepo עם `apps/web` ו-`apps/agent`.
  • Next.js TypeScript ל-Web.
  • Node.js TypeScript לסוכן המקומי.
  • קבצי `.env.example` ללא סודות.

תוצר

  • אפשר להריץ Web מקומי.
  • אפשר להריץ Agent bootstrap.
  • ה-email-to-print נשאר כ-Legacy side feature.
Phase 1

Database Design

תכולה

  • `print_jobs`, `print_agents`, `app_settings`.
  • Bucket פרטי: `print-files`.
  • RLS בסיסי ו-RPCs.
  • `status_token` לצפייה ציבורית בטוחה.

החלטות

  • ה-Agent משתמש ב-service role key.
  • Claim נעשה אטומית עם `FOR UPDATE SKIP LOCKED`.
  • קבצים לא ציבוריים, רק signed URL כשצריך.
Phase 2

Web App MVP

מסכים

  • עמוד העלאה ציבורי בעברית.
  • תצוגה מקדימה של PDF בדפדפן אם אפשר.
  • עמוד סטטוס עם token.
  • אדמין: תור, פרטי עבודה, הגדרות וסוכנים.

ולידציה

  • PDF בלבד וקובץ יחיד.
  • מגבלת גודל ועותקים.
  • שגיאות ברורות ולא טכניות.
Phase 3

API Routes

Endpoints

  • `POST /api/jobs/create`.
  • `GET /api/jobs/status`.
  • פעולות אדמין: approve, reject, cancel, retry, mark-printed.

אבטחה

  • service role רק בשרת.
  • אדמין נבדק גם לפי Supabase Auth וגם לפי `ADMIN_EMAILS`.
  • אין חשיפת URL לקובץ בעמוד הסטטוס.
Phase 4

Local Print Agent

זרימה

  • Heartbeat ל-`print_agents`.
  • Claim לעבודה אחת בכל פעם.
  • הורדת PDF ל-`DOWNLOAD_DIR`.
  • הדפסה דרך SumatraPDF.

בדיקה

  • `AGENT_DRY_RUN=true` לסימולציה.
  • שגיאות נשמרות ב-`error_message` ו-`agent_log`.
  • כיבוי מסודר מעדכן `offline`.
Phase 5

Security & Abuse

הגנות

  • PDF בלבד, גודל מוגבל, עותקים מוגבלים.
  • קוד העלאה אופציונלי למשתמשי המרחב.
  • Rate limiting במבנה ברור להמשך.

פרטיות

  • Storage פרטי.
  • Signed URLs קצרים לאדמין.
  • תכנון ניקוי קבצים ישנים.
Phase 6

Approval Mode

שני מצבים

  • Auto-print: העלאה יוצרת עבודה `approved`.
  • Manual approval: העלאה יוצרת עבודה `pending`.

ברירת מחדל

  • לפריסה אמיתית ראשונה מומלץ להתחיל ב-manual approval.
  • אחרי אמון ותפעול יציב אפשר לעבור לאוטומטי.
Phase 7

Local Testing

בדיקות Web

  • PDF תקין, קובץ לא PDF, גדול מדי, חסר שם.
  • סטטוס עם token נכון ושגוי.
  • פעולות אדמין: approve, cancel, retry.

בדיקות Agent

  • חסר SumatraPath נכשל ברור.
  • Heartbeat, claim, download, dry-run, printed.
  • כשל הדפסה מעדכן `failed`.
Phase 8

Deployment

ענן

  • Supabase project, migration, bucket פרטי.
  • Vercel ל-`apps/web`.
  • env vars מלאים ובדיקת העלאה.

מחשב הדפסה

  • Node.js LTS ו-SumatraPDF.
  • מדפסת מותקנת ב-Windows.
  • הרצה כ-Task Scheduler או NSSM.
Phase 9

Future Features

לא לבנות עכשיו

  • תשלום, מכסות, QR, release code.
  • כמה מדפסות, בחירת מדפסת, חיוב לפי עמוד.
  • המרת Word/PPT/תמונות ל-PDF.

הרחבות

  • התראות כשל, סריקת וירוסים, מחיקה אוטומטית.
  • ממשק עברי מלא ומיתוג white-label.
  • זיהוי יכולות מדפסת אמיתי לצבע/דו-צדדי.

פשוט

חוויה של העלאה מהירה למשתמשי קצה

יתרונות

  • מתאים למשתמשים לא טכניים.
  • מקטין טעויות בהעלאה.

חסרונות

  • פחות מידע על התור.
  • דורש מסך אדמין נפרד וחזק.

תפעולי

דשבורד צפוף וברור למנהל המרחב

יתרונות

  • מעולה לניהול תורים וכשלים.
  • הרבה מידע במסך אחד.

חסרונות

  • פחות נעים למשתמש מזדמן.
  • דורש סדר ויזואלי מאוד מוקפד.

שקוף

כל שלב מוצג למשתמש ולאדמין

יתרונות

  • בונה אמון סביב מסמכים רגישים.
  • קל לתמיכה ואבחון תקלות.

חסרונות

  • יותר טקסט ומידע במסכים.
  • דורש לוגים מדויקים מה-Agent.

מהיר

פעולות מיידיות וסטטוסים גדולים

יתרונות

  • טוב לסביבה עם הרבה תנועה.
  • מאפשר פעולות אדמין מהירות.

חסרונות

  • צריך הגנות מפני לחיצות שגויות.
  • פחות רגוע למסמכים רגישים.
המלצה למימוש הקרוב: להתחיל בכיוון "פשוט" עבור המשתמשים ובכיוון "תפעולי" עבור האדמין. כך מקבלים חוויית העלאה נקייה מצד אחד וניהול תור חזק מצד שני.
